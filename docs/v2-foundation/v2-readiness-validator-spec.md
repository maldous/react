# V2-Readiness Validator — Specification

> Governing intent: V1 (`maldous/react`, audited commit
> `918cd148569f6473eeaa58284933abdc0fe5bafe`) is the complete reference for V2; V2
> reuses/moves/refines/wraps proven V1 assets with **zero discovery** and a **clean final tree**.
> This spec describes `tools/v2-readiness/`, which **is implemented** (`src/index.mjs`) and wired as
> the npm scripts `v2:readiness` / `v2:readiness:json`; formal assurance artefacts are
> regenerated by `v2:formal-assurance`, and Universal Service Foundation assurance artefacts by
> `v2:usf-assurance`. It is **not** wired into `make all`; it runs
> on demand as the §0 / §15 gate of the branch-cut runbook. The validator writes no runtime file.

## 1. Purpose

A deterministic, read-only checker that proves `docs/v2-foundation/` planning closure is **total and
semantically self-consistent** — not merely count-reconciled. It fails closed: any contradiction is an
error and exit `1`.

## 2. Placement & invocation

- Location: `tools/v2-readiness/` (sibling of `tools/architecture/`).
  - `src/index.mjs` — CLI (`node src/index.mjs [--strict] [--json] [--repo <root>] [--pinned <sha>] [--historical] [--require-clean]`).
  - `src/load.mjs` — artefact + repo loaders; `src/package-status.mjs` — live package-removal engine + canonical-import scan.
  - `src/formal-assurance.mjs` — constructs the formal semantic graph, state-machine system,
    constraint reports, regeneration report, and entropy report.
  - `src/usf-assurance.mjs` — constructs Universal Service Foundation assurance graphs/reports for
    operational, observability, security, audit, event, environment, data, dependency, reliability,
    capability-coverage, and runtime-alignment domains.
  - `src/rules/*.mjs` — one module per rule (§4); `tests/*.test.mjs` — node:test suite (§5).
- Exit codes: `0` = all rules pass; `1` = one or more findings; `2` = bad input / missing artefact.
  `--strict` makes warnings fatal and enforces the cut-candidate sanity checks. `--historical`
  permits validating a non-HEAD snapshot. `--require-clean` additionally fails on a dirty working tree.
- **Not** referenced by any Makefile target. Wired only as root npm scripts `v2:readiness` /
  `v2:readiness:json`.

## 3. Canonical vocabularies (must match the artefacts exactly)

These are the ONLY allowed values; the validator normalises no aliases.

| Domain                | Canonical values                                                                                                                                               |
| --------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| file `disposition`    | `reuse-unchanged`, `git-move`, `split`, `merge`, `regenerate`, `archive-evidence`, `delete-after-proof`, `refactor-behind-contract`, `replace-retain-contract` |
| command `disposition` | `carry`, `merge`, `retire`                                                                                                                                     |
| test `migrationType`  | `carry`, `retarget`, `promote-to-conformance`, `retire`                                                                                                        |
| capability `status`   | `delivered-and-proven`, `not-applicable-final`, `rejected-final`, `superseded-by-proven-canonical`, `requires-v1-completion`                                   |

**Two commit concepts (§1):** `auditBaseCommit` = `918cd148569f6473eeaa58284933abdc0fe5bafe`
(immutable; the path-map/inventory describe this snapshot, preserving the original audit lineage).
`cutCandidateCommit` = the SHA the cut will freeze: defaults to HEAD, or `--pinned <sha>` at cut time;
under `--strict` it must resolve and (without `--historical`) equal HEAD. The cut-candidate file set is
read from the tracked working tree (`git ls-files`). `{{PINNED_V1_COMMIT}}` remains the sanctioned
named parameter in prose (not a forbidden placeholder).

**Two new structured artefacts:** `v1-post-audit-delta.json` (every file added/deleted since the audit
base, with introducing/removal commit, purpose, disposition, target, protecting tests, decision refs)
and `v1-completion-actions.json` (every V1C action + sub-action with id/parent/status/decision/ports/
contracts/persistence/permissions/audit/readiness/tests/UI-semantics/stop/V2-assets). Structured
package-removal evidence lives at `docs/evidence/lifecycle/removal/<pkg>/removal-evidence.json`.

Canonical import analysis (§5): both this validator (`package-status.collectImportMap`) and
`tools/architecture/validate-source-imports` consume the same TypeScript-AST primitive
`tools/architecture/_shared/import-edges.mjs` (`extractImports`) — covering static/export-from/
import-equals/type-only/dynamic/require/subpath — so the two tools can never drift to different
definitions of package consumption. A characterization test (`import-parity.test.mjs`) proves parity.

All audit surfaces are now examined by a live rule (`unexaminedAuditSurfaces = []`): file coverage
(R10), commands (R11), tests/proofs (R12), decisions/governance (R13), foundation artefacts (R14),
app roots (R15), services/clickthrough/SSO (R16), migrations/data (R17), environment/config (R18),
executables/Terraform/Playwright (R19), UI semantic harnesses (R20), semantic completeness (R22),
proof strength classification (R23), environment/cross-capability/event/operational/source/gate
semantics (R24-R29), formal mathematical assurance (R30-R37), and Universal Service Foundation assurance (R40-R50).

Report shape (§4): the validator emits `consistencyFindings` (must be 0), `completionBlockers` (array
of subjects) + `completionBlockerCount`, `unexaminedAuditSurfaces` (must be 0), and `cutReady`. A
green report means consistency findings, completion blockers, and unexamined audit surfaces are all
zero.

Scoping note: R1/R3 deliberately scan only the **closure-claim** artefacts, not this spec or the
runbook (which legitimately name the tokens they forbid). R2's field checks apply only to
`delivered-and-proven` records — a `requires-v1-completion` record is _expected_ to carry `missing` /
`not-yet-proven` honestly and must not be flagged. R1–R8 + R10–R50 together constitute the
**complete consistency validation**; R9 is the separate fail-closed cut gate.

## 5. Tests (`tools/v2-readiness/tests/`)

node:test, fixture-driven, no network/DB (`fixtures.mjs` provides a `cleanCtx()` factory that passes
all rules; each test clones + mutates it to fire one rule). The suite is registered in the
canonical `test:architecture` gate (ADR-ACT-0292), not only the ad-hoc `npm run v2:readiness`.

- `rules.test.mjs` — pass + targeted fail coverage across R1-R50, including: R9 reports completions + **live-present**
  packages + open decisions as blockers AND clears a package whose live status is removed even though
  its permanent path-map record remains; R10 file-set mismatch; R11 uncatalogued/stale command;
  R12 uninventoried/dangling test; R13 missing lineage / unknown action ref; R14 missing foundation
  artefact; R15 `apps/api` reference.
- `package-status.test.mjs` — the five required R9 package scenarios on real temp dirs:
  (1) package exists → blocker; (2) absent + valid evidence → clean; (3) absent + loader alias →
  blocker; (4) absent + tsconfig reference → blocker; (5) absent without evidence → blocker.
- `cli.test.mjs` — exit `0` path (clean context → zero findings, and live repo cut-ready JSON shape)
  / `2` (missing repo);
  `--json` report shape (`ok`, `findings`, `pinnedV1Commit`, `totalRules`).
- `golden.test.mjs` — runs against the real `docs/v2-foundation/` set; asserts zero consistency
  violations and that R9 matches the **exact derived blocker set** by subject and count.

## 6. Output
